# 运行时资产完整性检查清单

本文用于在 Pek.Skills 仓库内快速判断：哪些文件会直接影响安装和 Copilot 实际使用，哪些文件只是上游来源证据或分析资料，不应按“运行时缺失文件”处理。

## 1. 结论先行

- **会影响实际使用的，是运行时资产目录与安装脚本。**
- **不会直接影响实际使用的，是 docs 与 references 中保留的上游来源路径。**
- 判断仓库是否“基本可完全正常使用”，应优先检查安装、分发、发现和关键引用闭环，而不是先扫历史分析文档里的外部路径。

## 2. 必检目录

以下目录或文件缺失时，会直接影响资产安装或 Copilot 发现：

| 路径 | 必须性 | 作用 |
|------|------|------|
| `.github/copilot-instructions.md` | 必须 | 组织级全局协作指令源文件 |
| `.github/skills/` | 必须 | Skills 主体目录，要求 `<name>/SKILL.md` 结构 |
| `.github/instructions/` | 必须 | 模块指令目录，要求 `*.instructions.md` |
| `.github/prompts/` | 建议 | 提示词目录，要求 `*.prompt.md` |
| `.github/agents/` | 建议 | 代理目录，要求 `*.agent.md` |
| `scripts/install-copilot-assets.ps1` | 必须 | 主安装脚本 |
| `scripts/sync-skills-to-user.ps1` | 建议 | 旧入口兼容包装器 |

## 3. 关键结构要求

### 3.1 Skills

- 每个技能目录都应满足 `.github/skills/<name>/SKILL.md`
- 可附带 `references/` 等辅助资料目录
- 目录名建议与 frontmatter 中的 `name` 保持一致或语义接近

### 3.2 Instructions

- 文件名必须为 `*.instructions.md`
- 若在全局指令或其他指令中被显式引用，该文件必须真实存在
- 涉及 `applyTo` 的文件要确保 frontmatter 合法

### 3.3 Agents

- 文件名必须为 `*.agent.md`
- 若 README、文档或其他资产中宣称存在某个 agent，仓库中应有对应文件
- agent 文案应明确“目标代码仓库”和“Pek.Skills 资产仓库”的边界

### 3.4 Prompts

- 文件名必须为 `*.prompt.md`
- 若 docs 或 source-map 中记录了 prompt 来源，优先使用当前真实文件名，不要保留旧名造成错链

## 4. 安装后验证清单

运行以下命令：

```powershell
.\scripts\install-copilot-assets.ps1
```

验证点：

- 是否成功输出 Skills / Instructions / Prompts / Agents / 全局指令五个步骤
- 是否成功写入 `%APPDATA%\Code\User\prompts\skills\`
- 是否成功写入 `%APPDATA%\Code\User\prompts\*.instructions.md`
- 是否成功写入 `%APPDATA%\Code\User\prompts\*.agent.md`
- 是否成功生成 `peikesmart-global.instructions.md`

兼容入口验证：

```powershell
.\scripts\sync-skills-to-user.ps1
```

要求：旧脚本名也能正常转调主安装流程。

## 5. 阻塞性缺口判定

以下问题应视为阻塞性缺口，必须优先修复：

- 主指令或 README 引用了不存在的 `*.instructions.md`、`*.agent.md`、`*.prompt.md`
- 安装脚本引用了不存在的源目录或目标文件名
- README 中指导用户执行的脚本在 `scripts/` 中不存在
- 技能目录不符合 `<name>/SKILL.md` 结构，导致无法被正确复制或发现

## 6. 非阻塞问题判定

以下问题通常不影响运行时使用，但会影响维护体验：

- docs 中保留的上游 NewLife 路径，如 `d:\X\NewLife.Core`、`Doc/...`
- references 中的历史来源链接或模板路径
- 分析文档里的旧仓库名称、旧文件名、旧计划路径
- README 中统计数字过期，但不影响实际安装逻辑

这类问题建议逐步清理，但不应和运行时资产缺失混为一谈。

## 7. 推荐维护顺序

每次发版或大改前，建议按以下顺序自查：

1. 检查 `.github/skills/`、`.github/instructions/`、`.github/prompts/`、`.github/agents/` 是否完整
2. 检查 `install-copilot-assets.ps1` 是否仍能成功执行
3. 检查 README 中的脚本名、数量、文件列表是否与实际一致
4. 检查显式引用的指令、代理、提示词文件是否真实存在
5. 最后再整理 docs 与 references 中的历史来源说明

## 8. 当前仓库的适用解释

对 Pek.Skills 而言：

- `.github/` 与 `scripts/` 是运行时资产核心区域
- `docs/` 是组织说明、分析过程和维护资料区域
- `.github/skills/**/references/` 主要是来源证据与补充资料，不应默认按本地运行时资产处理

因此，当有人问“安装后能不能正常使用整个仓库的技能”时，判断重点应放在运行时资产核心区域，而不是历史来源文档是否全部能在本仓库本地闭环打开。
